# prompt-workshop 구조

> autoresearch가 LLM 학습을 자동 개선하듯,
> prompt-workshop은 <strong>업무 프롬프트를 자동 개선</strong>합니다.

## autoresearch와의 대응

[prompt-workshop](https://github.com/effectiveai-io/prompt-workshop)은
autoresearch와 같은 구조를 업무 프롬프트에 적용한 저장소입니다.

| autoresearch | prompt-workshop |
|---|---|
| `program.md` — AI의 연구 방향 지시 | `program.md` — AI의 프롬프트 개선 방향 지시 |
| `train.py` — AI가 수정하는 코드 | `prompts/*.md` — AI가 개선하는 프롬프트 |
| `prepare.py` — 고정된 평가 함수 | `evaluation/*.md` — 고정된 평가 기준 |
| `val_bpb` — 단일 지표 | 체크리스트 점수 (0~10점) |

## 저장소 구조

```
prompt-workshop/
├── program.md              # AI에게 주는 메타 지시서 (이것만 다듬습니다)
├── prompts/                # AI가 개선하는 업무 프롬프트
│   ├── code-review.md      # 코드 리뷰
│   ├── test-case.md        # TC 설계
│   ├── spec-review.md      # 기획서 리뷰
│   └── document-draft.md   # 문서 초안 작성
├── evaluation/             # 고정된 평가 기준 (수정하지 않습니다)
│   ├── code-review.md
│   ├── test-case.md
│   ├── spec-review.md
│   └── document-draft.md
└── samples/                # 실습용 샘플 (수정하지 않습니다)
    ├── sample-code.md      # 출금 요청 처리 코드
    ├── sample-spec.md      # 자동 매수 예약 기획서
    └── sample-feature.md   # 출금 주소 화이트리스트 기능
```

## 작동 방식

`program.md`에 자동 개선 사이클의 규칙이 적혀 있습니다.
Gemini에 저장소를 연결하고 "program.md를 따라해줘"라고 하면:

1. **현재 프롬프트 읽기**: `prompts/*.md` 파일을 읽습니다
2. **테스트 실행**: 그 프롬프트로 `samples/`의 샘플에 대한 결과물을 생성합니다
3. **자체 채점**: `evaluation/`의 평가 기준으로 결과물을 채점합니다
4. **개선안 도출**: 점수가 낮은 항목을 분석하고 프롬프트 개선안을 제안합니다
5. **개선된 프롬프트 생성**: 제안을 반영한 새 버전을 작성합니다
6. **다음 사이클로**: 개선된 프롬프트로 1번부터 다시 시작합니다

## program.md의 개선 방향 원칙

`program.md`에는 AI가 프롬프트를 개선할 때 따를 원칙이 적혀 있습니다:

- **구체적인 것이 추상적인 것보다 낫다**: "좋은 리뷰를 해줘" (X) → "변수명이 camelCase인지 확인해줘" (O)
- **적은 규칙이 많은 규칙보다 낫다**: 20개의 약한 규칙보다 5개의 강한 규칙
- **예시가 있으면 더 좋다**: "이런 경우 이렇게 리뷰해줘"라는 예시가 포함되면 품질이 올라감
- **도메인 맥락이 가장 큰 점수 상승**: 일반 규칙보다 "우리 팀만의 규칙"이 점수를 크게 올림

<Callout type="info">
**실무에서의 핵심**

워크숍에서는 샘플 코드/기획서로 실습하지만, 돌아가서:
- `samples/`에 **실제 팀 코드/기획서**를 넣고
- `prompts/`를 **실제 팀 컨벤션**으로 수정하고
- 다시 돌리면

→ 팀 맞춤형 AI 프롬프트가 자동으로 만들어지는 구조가 됩니다.
</Callout>
