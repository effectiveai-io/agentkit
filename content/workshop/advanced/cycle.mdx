# 지침서 개선 사이클

## Step 1: 내 반복 업무 선택 (10분)

> "지난 2주간 AI에게 3번 이상 같은 맥락을 반복 설명한 적이 있다면, 그게 바로 지침서의 재료입니다."

### 직군별 반복 업무 예시

| 직군 | 반복 업무 |
|------|----------|
| BE 개발 | 코드 리뷰, 테스트 코드 작성, PR 설명 작성, 슬랙 문의 대응 |
| FE 개발 | 코드 리뷰, 피그마 속성을 CSS로 옮기기, 컴포넌트 작성 |
| PM/기획 | 기획서 초안, 요구사항 정리, 보고 문서 작성, 지라 검토 |
| QA | TC 설계, 기획서 리뷰, 검수, 결함 리포트 |
| UX | 디자인 산출물 제작, UX 정의, 개발 문의 대응 |

**소그룹(3~4명)에서 각자 선택한 업무를 공유하고, 하나를 확정합니다.**

## Step 2: 평가 기준 정의 (15분)

> autoresearch의 `val_bpb`에 해당하는 것.
> 지침서가 좋아지고 있는지 측정할 기준이 있어야 사이클이 의미있습니다.

각 항목을 0~2점으로 채점합니다 (0: 미반영, 1: 부분 반영, 2: 완전 반영).

### 평가 기준 예시

**코드 리뷰 지침서**
- [ ] 팀 컨벤션(네이밍, 구조)이 반영되었는가
- [ ] 보안/성능 관점의 리뷰가 포함되었는가
- [ ] 비즈니스 로직 맥락을 이해한 피드백인가
- [ ] 개선 제안이 구체적인가 (코드 예시 포함)
- → 만점: 8점

**기획서 리뷰 지침서**
- [ ] 누락된 예외 케이스를 짚어냈는가
- [ ] 기술적 실현 가능성 관점이 포함되었는가
- [ ] 사용자 시나리오 기반으로 검토했는가
- [ ] 다른 기획서와의 정합성을 확인했는가
- → 만점: 8점

**TC 설계 지침서**
- [ ] 정상/비정상 시나리오가 구분되었는가
- [ ] 경계값 테스트가 포함되었는가
- [ ] 우선순위가 반영되었는가
- [ ] 재현 가능한 형식인가
- → 만점: 8점

<Callout type="info">
**평가 기준 점검 질문**: "이 기준이면 정말 좋은 결과물과 나쁜 결과물을 구분할 수 있는가?"

소그룹에서 서로의 평가 기준을 피드백해주세요.
</Callout>

## 3사이클 실습 (50분)

<Callout type="warning">
**중요**: 매 사이클에서 <strong>AI에게 시키는 요청(프롬프트)은 동일하게 유지</strong>합니다.
변하는 것은 지침서뿐입니다. 이래야 "지침서가 좋아져서 결과가 나아졌다"를 분리할 수 있습니다.
</Callout>

**도구**: ChatGPT 커스텀 지침 (Settings → Customize ChatGPT) 또는 Gemini Gems

```
매 사이클:
1. 지침서 수정 → 커스텀 지침에 붙여넣기 (5분)
2. 동일한 업무 요청을 AI에게 시키기 (5분)
3. 평가 기준 체크리스트로 채점 + 개선 포인트 메모 (5분)
```

<Steps>

### Cycle 1 — 베이스라인 (15분)

**지침서 v0**: 빈 상태 또는 최소한의 한 줄

예: `"우리 팀 기준으로 코드 리뷰해줘"`

**AI에게 시킬 요청**: 강사가 제공한 샘플 시나리오를 사용합니다.

채점 후 점수를 기록하세요.

<Callout type="default">
**예상 결과**: 일반적이고 뻔한 AI 응답 → 낮은 점수

이 점수를 기억하세요. 사이클을 돌 때마다 올라가는 걸 직접 볼 겁니다.
</Callout>

### Cycle 2 — 컨벤션 주입 (15분)

**지침서 v1**: Cycle 1에서 빠졌던 <strong>팀 컨벤션</strong>을 추가합니다.

> 우아한형제들 개발자도 `copilot-instructions.md`에 팀 컨벤션을 넣었더니 코파일럿이 열일하기 시작했습니다.
> — [코파일럿 "열일"하게 만드는 방법](https://techblog.woowahan.com/21240/) (김민희, 2025.02)

**넣어야 하는 것**:
- AI가 코드를 읽어도 알 수 없는 것 (팀 컨벤션, 브랜드 가이드)
- 기본값과 다른 우리 팀만의 방식
- 구체적이고 검증 가능한 지침

**넣지 말아야 하는 것**:
- AI가 이미 아는 일반 규칙 ("클린 코드를 작성하라")
- 추상적인 지시 ("UI를 잘 만들어라")

**v1 예시 (코드 리뷰)**:

```markdown
## 코드 리뷰 기준

### 네이밍
- 변수명: camelCase, 헝가리안 표기법 금지
- Boolean: is/has/should 접두사 필수

### 구조
- DTO 변환은 서비스 레이어에서만
- 컨트롤러에 비즈니스 로직 금지

### 테스트
- given-when-then 패턴 필수
- 테스트 메서드명: should_기대결과_when_조건

### 리뷰 형식
- 심각도 표시: [MUST] / [SHOULD] / [NICE]
- 개선 제안 시 코드 예시 포함
```

같은 요청을 다시 시키고, 같은 체크리스트로 채점합니다.

<Callout type="default">
**예상 결과**: 컨벤션이 반영된 구체적 응답 → 점수 상승
</Callout>

### Cycle 3 — 도메인 맥락 주입 (15분)

**지침서 v2**: v1에 <strong>비즈니스/도메인 맥락</strong>을 추가합니다.

> 테스트 자동화 플러그인이 성공한 이유는 "프로젝트의 실제 구조(PSI 트리)"를 AI에게 줬기 때문입니다.
> — [AI와 함께하는 테스트 자동화](https://techblog.woowahan.com/24568/) (강규한, 2025.12)

**v2에 추가할 것 (코드 리뷰 예시)**:

```markdown
### 도메인 맥락
- 금융 서비스: 금액 계산은 BigDecimal 필수
- 보안: 로그에 민감정보(계좌번호, 개인키) 포함 시 [MUST] 지적
- 동시성: 주문/거래 관련 코드는 동시성 이슈 반드시 검토

### 리뷰 우선순위
1. 보안·금액 정확성 (반드시 먼저)
2. 비즈니스 로직 정합성
3. 성능·동시성
4. 코드 스타일·가독성
```

같은 요청, 같은 체크리스트로 채점합니다.

<Callout type="default">
**예상 결과**: 도메인 특화 응답 → 추가 점수 상승
</Callout>

</Steps>

## 결과 정리 (5분)

3사이클의 점수를 기록합니다:

```
Cycle 1 (v0 — 빈 지침서):     __/8점
Cycle 2 (v1 — 컨벤션 주입):    __/8점
Cycle 3 (v2 — 도메인 맥락):    __/8점
```

<Callout type="info">
autoresearch는 밤새 100번 돌려서 모델을 개선합니다.
여러분은 3번 돌려서 AI의 업무 품질을 올렸습니다.
돌아가서 10번만 더 돌리면 어떻게 될까요?
</Callout>
