# 실습: 내 업무에 적용하기

<Timer minutes={25} />

<Callout type="default">
**이 단계의 목표**: Phase 2에서 경험한 구조를 내 업무에 적용하기

Phase 2에서는 프롬프트(지시서)를 다듬어서 AI가 <strong>코드</strong>를 자동 개선하게 했습니다.
이제 같은 구조를 내 업무에 적용합니다:
- PM이라면 → 프롬프트를 다듬어서 AI가 <strong>기획서</strong>를 개선하게
- QA라면 → 프롬프트를 다듬어서 AI가 <strong>테스트 케이스</strong>를 개선하게
- FE라면 → 프롬프트를 다듬어서 AI가 <strong>컴포넌트</strong>를 개선하게
</Callout>

## Step 1: 내 업무용 자동 개선 앱 만들기

Gemini에서 Canvas 모드를 켜고, 아래 프롬프트에서 `[...]` 부분을 본인 업무로 바꿔서 입력하세요.

<Tabs items={['내 업무용 앱 프롬프트', 'PM 예시', 'QA 예시', 'FE 예시']}>
<Tabs.Tab>

```
Gemini Canvas로 "[내 업무] 자동 개선기" 웹앱을 만들어줘.
내장된 Gemini 2.5 Flash 모델을 사용해줘.

이 앱은 사용자가 프롬프트(지시서)를 입력하면,
AI가 그 지시서를 따라 [개선 대상]을 리뷰하고 → 개선하고 → 채점하는 사이클을 반복하는 앱이야.

## 내 업무 설명
나는 [직군]이고, [구체적인 반복 업무]를 하고 있어.

## 이 업무의 평가 기준
이 업무의 결과물 품질을 측정할 수 있는 평가 기준을 만들어줘.
채점 원칙: 결과물에서 세거나 확인할 수 있는 기준으로 채점.

## 화면 구성

### 개선 대상 미리보기
- 개선할 [결과물]을 표시

### 입력 영역
- "프롬프트 (지시서)" 텍스트 영역 — AI가 [결과물]을 리뷰할 때 따를 지침
- "사이클 수" 선택 (1~5, 기본값 3)
- "자동 개선 시작" 버튼

### 결과 영역
- 사이클별 품질 점수 차트
- 각 사이클 상세:
  - 리뷰 결과
  - 개선된 [결과물] (이전 대비 변경점 하이라이트)
  - 채점 결과 + 채점 근거
  - keep/discard 판단 (점수가 내려가면 이전 버전으로 되돌림)
- 최종 개선된 [결과물] 복사 버튼

## 동작 방식 — 리뷰와 채점의 엄격한 분리

리뷰 단계에서는 평가 기준을 모르는 것처럼 행동합니다.
오직 프롬프트(지시서)의 지시만 따라 [결과물]을 리뷰합니다.
채점은 개선이 완료된 후에만 시작됩니다.

"자동 개선 시작"을 누르면:
1. [리뷰] 프롬프트(지시서)만 참고해서 현재 [결과물]을 리뷰
2. [개선] 리뷰 결과를 반영해서 [결과물]을 개선
3. [채점] 평가 기준으로 개선된 [결과물]을 채점
4. [판단] 이전보다 올랐으면 keep, 내려갔으면 discard (이전 버전으로 되돌림)
5. [반복] keep된 [결과물]을 대상으로 1번부터 반복
```

## Step 2: 내 업무 샘플 준비하기

앱이 만들어지면, AI가 개선할 <strong>입력물(코드, 문서, 기획서 등)</strong>을 준비합니다.

### 방법 A: 실제 업무 자료가 있다면

| 직군 | 가져올 수 있는 것 예시 |
|------|---------------------|
| BE 개발 | 최근 PR의 코드, API 스펙 문서 |
| FE 개발 | 최근 작성한 컴포넌트 코드, CSS 파일 |
| PM | 최근 작성한 기획서, 요구사항 문서 |
| QA | 최근 테스트한 기능 설명서, 결함 리포트 |

### 방법 B: 지금 당장 가져오기 어렵다면

Gemini에게 만들게 합니다:

```
나는 [직군]이고, [구체적인 업무 설명]을 반복적으로 하고 있어.
AI가 개선할 수 있는 현실적인 [결과물] 샘플을 만들어줘.
겉보기에는 괜찮지만 개선할 여지가 있는 형태로.
```

## Step 3: 자동 개선 사이클 돌리기

1. 프롬프트(지시서)를 입력 — 최소한으로 시작 (예: "이 기획서를 리뷰해줘")
2. 샘플을 개선 대상 영역에 붙여넣기
3. <strong>"자동 개선 시작"</strong> 클릭
4. 사이클이 돌면서 결과물이 개선되는 것을 관찰
5. 정체되면 → **프롬프트(지시서)를 다듬어서 다시 돌리기**

> Phase 2에서 코드를 개선했던 것과 같은 경험입니다.
> 프롬프트(지시서)를 잘 쓸수록 AI가 결과물을 더 잘 개선합니다.
