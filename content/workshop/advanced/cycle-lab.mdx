# 실습: 코드 리뷰 프롬프트 개선

3단계로 점점 깊어집니다.

| 단계 | 하는 일 | 왜 하는지 |
|------|---------|----------|
| **1단계** | 저장소 연결 → 수동으로 리뷰 + 채점 | "내 프롬프트가 어느 수준인지" 베이스라인 확인 |
| **2단계** | 자동 개선 Canvas 앱 만들기 → 사이클 돌리기 | "매번 수동은 귀찮다" → 자동화 |
| **3단계** | 5개 코드로 검증 | "하나의 코드에만 맞춘 건 아닌지" 범용성 확인 |

## 1단계: 저장소 연결 + 수동 1사이클

<Timer minutes={10} />

<Callout type="default">
**이 단계의 목표**: 내 프롬프트의 현재 수준(베이스라인)을 확인하기

autoresearch에서도 첫 실행은 항상 train.py를 있는 그대로 돌려서 베이스라인을 잡습니다.
</Callout>

### 저장소 fork + Gemini 연결

1. [prompt-workshop](https://github.com/effectiveai-io/prompt-workshop) 저장소를 fork
2. Gemini 새 탭을 열고, 입력창 왼쪽의 **+** 버튼 → **코드 가져오기** 선택

![Gemini 도구 메뉴 — 코드 가져오기 선택](/workshop/advanced/gemini-tools-menu.png)

3. fork한 저장소 URL을 입력하고 **가져오기** 클릭

![코드 가져오기 — prompt-workshop 저장소 URL 입력](/workshop/advanced/gemini-import-prompt-workshop.png)

### 수동으로 한 번 돌려보기

```
이 저장소의 구조를 먼저 파악해줘.

그리고 prompts/code-review.md를 프롬프트로 사용해서
samples/sample-code-1.md의 코드를 리뷰해줘.

리뷰 결과를 evaluation/code-review.md 기준으로 채점하고,
10개 항목별 점수와 채점 근거를 보여줘.
```

### 관찰하기

- **총점이 몇 점인지** 기억하세요 — 이게 베이스라인입니다
- 어떤 항목이 1~2점인지 확인 — 프롬프트에 해당 내용이 없다는 뜻
- "이 부분을 프롬프트에 추가하면 점수가 오를 텐데..." 생각이 드나요?

</Callout>

## 2단계: 자동 개선 Canvas 앱 만들기

<Timer minutes={20} />

<Callout type="default">
**이 단계의 목표**: 1단계에서 수동으로 한 "리뷰 → 채점 → 개선" 사이클을 AI가 자동으로 반복하게 만들기

매번 프롬프트를 고치고, 다시 리뷰 시키고, 다시 채점하는 건 귀찮습니다.
autoresearch처럼 <strong>버튼 하나로 AI가 알아서 사이클을 돌게</strong> 만듭니다.
</Callout>

같은 Gemini 대화에서 Canvas 모드를 켜고 아래 프롬프트를 입력하세요.
저장소가 이미 연결되어 있으므로, 평가 기준과 샘플 코드를 저장소에서 직접 참조합니다.

<Tabs items={['자동 개선 앱 프롬프트']}>
<Tabs.Tab>

```
이 저장소를 참고해서, Gemini Canvas로 "프롬프트 자동 개선기" 웹앱을 만들어줘.
내장된 Gemini 2.5 Flash 모델을 사용해줘.

이 앱은 사용자가 코드 리뷰 프롬프트를 입력하면,
자동으로 결과물 생성 → 채점 → 프롬프트 개선을 반복하는 앱이야.

## 저장소에서 가져올 파일

평가 기준:
- evaluation/code-review.md (10개 항목, 각 1~5점, 만점 50점 — 측정 가능한 채점 기준)

테스트 코드:
- samples/sample-code-1.md (기본 테스트 코드로 사용)

## 화면 구성

### 입력 영역
- "프롬프트" 텍스트 영역 (코드 리뷰 지침서를 입력)
- "사이클 수" 선택 (1~5, 기본값 3)
- "자동 개선 시작" 버튼

### 결과 영역

#### 상단: 사이클별 총점 바 차트
- 50점 만점, v0→v1→v2... 점수 변화를 시각화
- 현재 진행 중인 사이클 표시

#### 하단: 사이클별 상세 (탭 또는 아코디언)
각 사이클(v0, v1, v2...)을 클릭하면 아래 4개 영역을 보여줌:

**A. 이번 사이클에서 사용한 프롬프트**
- 프롬프트 전문을 코드 블록으로 표시
- 이전 버전과 달라진 부분을 하이라이트 (추가된 줄은 초록, 삭제된 줄은 빨간)

**B. AI가 생성한 코드 리뷰 결과물**
- 이 프롬프트로 실제 생성된 코드 리뷰 전문

**C. 채점 결과**
- 10개 항목별 점수표:
  | 항목 | 점수 | 이전 대비 | 채점 근거 |
  - 채점 근거: 왜 이 점수인지 구체적 이유
- 총점 + 이전 사이클 대비 변화량

**D. 개선 분석**
- 이번에 점수가 오른 항목: 뭘 바꿔서 올랐는지
- 여전히 낮은 항목: 왜 낮은지 + 다음에 어떻게 고칠지

#### 최종 결과
- 최종 프롬프트 복사 버튼
- 전체 사이클 점수 변화 요약

## 동작 방식

"자동 개선 시작"을 누르면 내장된 Gemini 2.5 Flash 모델이 자동으로:
1. 현재 프롬프트로 테스트 코드에 대한 코드 리뷰 생성
2. 저장소의 evaluation/code-review.md 기준으로 채점 + 채점 근거 기록
3. 점수가 낮은 항목 분석 → 프롬프트 개선안 도출
4. 개선된 프롬프트 생성 (이전 버전과의 차이를 기록)
5. 개선된 프롬프트로 1번부터 반복
매 사이클 완료 시 바 차트 + 상세 결과가 업데이트됨.

## 디자인
- 한국어 UI
- 다크 모드
- 바 차트 색상: 15점 이하 회색, 16~30 파란색, 31 이상 초록색
```

### 앱이 만들어지면

1. 1단계에서 사용한 프롬프트(`prompts/code-review.md`)를 앱에 붙여넣기
2. <strong>"자동 개선 시작"</strong> 클릭

### 1단계 점수와 비교하기

- **가장 크게 오른 항목**: AI가 어떤 규칙을 추가했는지 확인
- **여전히 낮은 항목**: 방향을 수정해서 다시 돌려볼 수 있음

## 3단계: 5개 코드로 범용성 검증

<Timer minutes={15} />

<Callout type="default">
**이 단계의 목표**: 2단계에서 개선된 프롬프트가 "다른 코드에도 통하는지" 검증하기

시험공부에 비유하면 이렇습니다:
- 2단계는 <strong>연습문제로 공부한 것</strong>입니다 — sample-code-1로 개선하고, sample-code-1로 채점했으니 점수가 잘 나옵니다.
- 3단계는 <strong>처음 보는 시험 문제로 실력을 확인하는 것</strong>입니다 — 나머지 4개 코드로 테스트해서 진짜 실력을 봅니다.

autoresearch에서도 같은 원리입니다. AI가 코드를 개선할 때 쓴 데이터가 아니라,
<strong>한 번도 본 적 없는 별도의 데이터</strong>(`val_bpb`)로 측정합니다.
연습문제만 잘 푸는 건지, 진짜 실력이 있는 건지 구분하기 위해서입니다.
</Callout>

같은 Gemini 대화에서 Canvas로 검증 앱을 만듭니다:

<Tabs items={['검증 앱 프롬프트']}>
<Tabs.Tab>

```
이 저장소를 참고해서, Gemini Canvas로 "프롬프트 검증기" 웹앱을 만들어줘.
내장된 Gemini 2.5 Flash 모델을 사용해줘.

이 앱은 하나의 코드 리뷰 프롬프트가 여러 코드에서 얼마나 잘 작동하는지 검증하는 앱이야.

## 저장소에서 가져올 파일

평가 기준:
- evaluation/code-review.md

검증할 코드 (5개):
- samples/sample-code-1.md (출금 요청 처리)
- samples/sample-code-2.md (자동 매수 예약)
- samples/sample-code-3.md (주소 화이트리스트)
- samples/sample-code-4.md (거래 내역 조회/CSV)
- samples/sample-code-5.md (알림 설정/발송)

## 화면 구성

### 1. 소스 코드 미리보기 영역
- 저장소에서 가져온 5개 코드를 탭으로 보여주기
  (출금 | 자동매수 | 화이트리스트 | 거래내역 | 알림)
- 각 탭에 해당 코드의 내용과 "참고 맥락"을 표시

### 2. 프롬프트 입력 영역
- "코드 리뷰 프롬프트" 텍스트 영역
- "검증 시작" 버튼

### 3. 진행 상황 영역
- "검증 시작"을 누르면 5개 코드를 순서대로 분석
- 각 코드 분석 시 스텝 바이 스텝으로 보여주기:
  - Step 1/3: "출금 코드 리뷰 생성 중..." → 완료 시 ✅
  - Step 2/3: "evaluation/code-review.md 기준으로 채점 중..." → 완료 시 ✅ + 점수
  - Step 3/3: "다음 코드로 이동..." → 완료 시 ✅
- 현재 진행 중인 코드 프로그레스 (예: 2/5)

### 4. 결과 영역

#### 4-1. 총점 비교 바 차트
- 5개 코드별 총점 바 차트 (가로 배치, 50점 만점)
- "평균 점수"와 "편차" 표시

#### 4-2. 항목별 점수 히트맵
- 10행(평가 항목) x 5열(코드) 테이블
- 각 셀에 점수 + 배경색 (1~2 빨간, 3 노란, 4~5 초록)
- 행 끝에 해당 항목의 평균 점수

#### 4-3. 코드별 상세 평가 (아코디언)
- AI가 생성한 코드 리뷰 전문
- 10개 항목별 점수 + 채점 근거
- 발견한 이슈 / 놓친 이슈

#### 4-4. 종합 분석
- 강점 / 약점 / 개선 제안
- 최종 프롬프트 복사 버튼

## 디자인
- 한국어 UI, 다크 모드
- 바 차트: 15점 이하 회색, 16~30 파란색, 31 이상 초록색
- 진행 중 파란 하이라이트, 완료 초록 체크
```

</Tabs.Tab>
</Tabs>

### 검증하기

1. 2단계에서 만든 최종 프롬프트를 앱에 붙여넣기
2. <strong>"검증 시작"</strong> 클릭
3. 5개 코드에 대한 점수 비교를 확인

### 관찰하기

- **바 차트가 고르면**: 프롬프트가 범용적으로 잘 작동하는 것
- **특정 코드만 점수가 낮으면**: 프롬프트가 특정 패턴에만 맞춰진 것
- **히트맵에서 빨간 셀이 많은 항목**: 프롬프트에서 해당 관점이 약하다는 뜻
- **편차가 크면**: 프롬프트를 보강한 뒤 다시 검증

> **이것이 autoresearch에서 validation set의 역할입니다.**
> 하나의 코드에만 잘 맞는 프롬프트가 아니라, 어떤 코드에도 통하는 프롬프트를 만드는 것.
