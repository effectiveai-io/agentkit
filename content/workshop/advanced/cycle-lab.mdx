# 실습: 코드 리뷰 프롬프트 개선

<Timer minutes={30} />

3단계로 점점 깊어집니다. 각자 속도에 맞춰 진행하세요.

| 단계 | 하는 일 | 왜 하는지 |
|------|---------|----------|
| **1단계** | 저장소 연결 → 수동으로 리뷰 + 채점 | "내 프롬프트가 어느 수준인지" 베이스라인 확인 |
| **2단계** | 자동 개선 Canvas 앱 만들기 → 사이클 돌리기 | "매번 수동은 귀찮다" → 자동화 |
| **3단계** | 같은 앱에서 다른 코드로 돌려보기 | "이 프롬프트가 다른 코드에도 통하는지" 검증 |

---

## 1단계: 저장소 연결 + 수동 1사이클

<Callout type="default">
**이 단계의 목표**: 내 프롬프트의 현재 수준(베이스라인)을 확인하기

autoresearch에서도 첫 실행은 항상 train.py를 있는 그대로 돌려서 베이스라인을 잡습니다.
</Callout>

### 저장소 fork + Gemini 연결

1. [prompt-workshop](https://github.com/effectiveai-io/prompt-workshop) 저장소를 fork
2. Gemini 새 탭을 열고, 입력창 왼쪽의 **+** 버튼 → **코드 가져오기** 선택

![Gemini 도구 메뉴 — 코드 가져오기 선택](/workshop/advanced/gemini-tools-menu.png)

3. fork한 저장소 URL을 입력하고 **가져오기** 클릭

![코드 가져오기 — prompt-workshop 저장소 URL 입력](/workshop/advanced/gemini-import-prompt-workshop.png)

### 수동으로 한 번 돌려보기

```
이 저장소의 구조를 먼저 파악해줘.

그리고 prompts/code-review.md를 프롬프트로 사용해서
samples/sample-code-1.md의 코드를 리뷰해줘.

리뷰 결과를 evaluation/code-review.md 기준으로 채점하고,
10개 항목별 점수와 채점 근거를 보여줘.
```

### 관찰하기

최소 프롬프트("코드를 리뷰해주세요")로 시작하면, 대략 이런 결과가 나옵니다:

| 유형 | 해당 항목 | 예상 결과 | 이유 |
|------|----------|----------|------|
| **AI가 원래 잘하는 것** | 보안, 금액 정밀도, 동시성, 에러 처리 | 중간 점수 | 프롬프트 없이도 AI가 기본적으로 찾아냄 |
| **프롬프트가 있어야 하는 것** | 네이밍, 레이어, 테스트, 심각도, 코드 예시, 우선순위 | 최하점 | 프롬프트에 해당 지침이 없으면 AI가 아예 안 함 |

**총점은 낮게 나옵니다.** 여기서 확인할 것:

- **최하점인 항목** — 이것들이 자동 개선으로 올라갈 여지가 있는 항목
- **중간 점수인 항목** — AI가 기본으로 찾지만, 프롬프트에 구체적 지침을 넣으면 높은 점수까지 올라감
- 2단계에서 자동 개선을 돌리면, 최하점 → 중간, 중간 → 높은 점수로 올라가면서 총점이 크게 상승

<Callout type="info">
"1점인 항목이 이렇게 많네" — 바로 그걸 자동으로 채워주는 앱을 다음 단계에서 만듭니다.
</Callout>

## 2단계: 자동 개선 Canvas 앱 만들기

<Callout type="default">
**이 단계의 목표**: 1단계에서 수동으로 한 "리뷰 → 채점 → 개선" 사이클을 AI가 자동으로 반복하게 만들기

매번 프롬프트를 고치고, 다시 리뷰 시키고, 다시 채점하는 건 귀찮습니다.
autoresearch처럼 <strong>버튼 하나로 AI가 알아서 사이클을 돌게</strong> 만듭니다.
</Callout>

같은 Gemini 대화에서 **도구 → Canvas**를 선택해서 Canvas 모드를 켜고, 아래 프롬프트를 입력하세요.
저장소가 이미 연결되어 있으므로, 평가 기준과 샘플 코드를 저장소에서 직접 참조합니다.

![대화 중 도구 메뉴에서 Canvas 선택](/workshop/advanced/gemini-canvas-in-chat.png)

<Tabs items={['자동 개선 앱 프롬프트']}>
<Tabs.Tab>

```
이 저장소를 참고해서, Gemini Canvas로 "프롬프트 자동 개선기" 웹앱을 만들어줘.
내장된 Gemini 2.5 Flash 모델을 사용해줘.

이 앱은 사용자가 코드 리뷰 프롬프트를 입력하면,
자동으로 결과물 생성 → 채점 → 프롬프트 개선을 반복하는 앱이야.

## 저장소에서 가져올 파일

평가 기준:
- evaluation/code-review.md의 내용을 그대로 따를 것 (항목 수, 점수 범위, 채점 근거 모두 이 파일 기준)

테스트 코드:
- samples/sample-code-1.md (기본 테스트 코드로 사용)

## 화면 구성

### 입력 영역
- "프롬프트" 텍스트 영역 (코드 리뷰 지침서를 입력)
- "사이클 수" 선택 (1~5, 기본값 3)
- "자동 개선 시작" 버튼

### 결과 영역

#### 상단: 사이클별 총점 바 차트
- evaluation/code-review.md 기준 만점, v0→v1→v2... 점수 변화를 시각화
- 현재 진행 중인 사이클 표시

#### 하단: 사이클별 상세 (탭 또는 아코디언)
각 사이클(v0, v1, v2...)을 클릭하면 아래 4개 영역을 보여줌:

**A. 이번 사이클에서 사용한 프롬프트**
- 프롬프트 전문을 코드 블록으로 표시
- 이전 버전과 달라진 부분을 하이라이트 (추가된 줄은 초록, 삭제된 줄은 빨간)

**B. AI가 생성한 코드 리뷰 결과물**
- 이 프롬프트로 실제 생성된 코드 리뷰 전문

**C. 채점 결과**
- 10개 항목별 점수표:
  | 항목 | 점수 | 이전 대비 | 채점 근거 |
  - 채점 근거: 왜 이 점수인지 구체적 이유
- 총점 + 이전 사이클 대비 변화량

**D. 개선 분석**
- 이번에 점수가 오른 항목: 뭘 바꿔서 올랐는지
- 여전히 낮은 항목: 왜 낮은지 + 다음에 어떻게 고칠지

#### 최종 결과
- 최종 프롬프트 복사 버튼
- 전체 사이클 점수 변화 요약

## 동작 방식

"자동 개선 시작"을 누르면 내장된 Gemini 2.5 Flash 모델이 자동으로:
1. 현재 프롬프트로 테스트 코드에 대한 코드 리뷰 생성
2. 저장소의 evaluation/code-review.md 기준으로 채점 + 채점 근거 기록
3. 점수가 낮은 항목 분석 → 프롬프트 개선안 도출
4. 개선된 프롬프트 생성 (이전 버전과의 차이를 기록)
5. 개선된 프롬프트로 1번부터 반복
매 사이클 완료 시 바 차트 + 상세 결과가 업데이트됨.

## 디자인
- 한국어 UI
- 다크 모드
- 바 차트 색상: 점수가 낮으면 회색, 중간이면 파란색, 높으면 초록색
```

</Tabs.Tab>
</Tabs>

### 앱이 만들어지면

1. 1단계에서 사용한 프롬프트(`prompts/code-review.md`)를 앱에 붙여넣기
2. <strong>"자동 개선 시작"</strong> 클릭
3. AI가 사이클을 돌면서 바 차트가 올라가는 것을 관찰
4. 각 사이클의 상세를 열어서 "AI가 뭘 바꿨는지" 확인

### 1단계 점수와 비교하기

자동 개선 전후로 어떤 변화가 있는지 확인합니다:

| 유형 | 1단계 (베이스라인) | 2단계 (자동 개선 후) | 어떤 변화가 있었나 |
|------|-----------------|-------------------|-----------------|
| 낮은 점수 항목 (네이밍, 심각도 등) | 최하점 | 중간 점수 | AI가 프롬프트에 해당 규칙을 추가함 |
| 중간 점수 항목 (보안, 금액 등) | 중간 점수 | 높은 점수 | AI가 더 구체적인 지침을 추가함 (코드 예시, 시나리오 등) |
| 총점 | 낮음 | 크게 상승 | 프롬프트가 구체적일수록 점수가 올라감 |

**확인할 것:**
- **가장 크게 오른 항목**: AI가 어떤 규칙을 추가했는지 각 사이클의 프롬프트 변경점을 열어 확인
- **여전히 낮은 항목**: 방향을 수정해서 다시 돌려볼 수 있음
- **프롬프트 변경점 하이라이트**: 초록색(추가)과 빨간색(삭제) 부분이 점수 변화의 원인

<Callout type="info">
앱이 완벽하지 않아도 됩니다.
핵심은 <strong>"프롬프트에 지침이 없으면 1점, 추가하면 3점, 구체적으로 하면 5점"</strong>이라는 패턴을 직접 확인하는 것입니다.
</Callout>

## 3단계: 다른 코드로 검증하기

<Callout type="default">
**이 단계의 목표**: 2단계에서 개선된 프롬프트가 "다른 코드에도 통하는지" 검증하기

시험공부에 비유하면 이렇습니다:
- 2단계는 <strong>연습문제로 공부한 것</strong>입니다 — sample-code-1로 개선하고, sample-code-1로 채점했으니 점수가 잘 나옵니다.
- 3단계는 <strong>처음 보는 시험 문제로 실력을 확인하는 것</strong>입니다 — 다른 코드로 테스트해서 진짜 실력을 봅니다.

autoresearch에서도 같은 원리입니다. AI가 코드를 개선할 때 쓴 데이터가 아니라,
<strong>한 번도 본 적 없는 별도의 데이터</strong>(`val_bpb`)로 측정합니다.
</Callout>

### 앱에 검증용 코드 추가하기

새 앱을 만들 필요 없습니다. 2단계에서 만든 앱에 샘플 코드를 추가합니다.
Gemini 대화에서 아래와 같이 요청하세요:

```
이 앱에 검증 기능을 추가해줘.

저장소의 나머지 4개 샘플 코드를 추가로 가져와서,
2단계에서 개선한 최종 프롬프트로 5개 코드 모두에 대해 리뷰 + 채점 결과를
비교할 수 있게 해줘.

추가할 코드:
- samples/sample-code-2.md (자동 매수 예약)
- samples/sample-code-3.md (주소 화이트리스트)
- samples/sample-code-4.md (거래 내역 조회/CSV)
- samples/sample-code-5.md (알림 설정/발송)

검증 결과 표시:
- 5개 코드별 총점 바 차트 (가로 배치, evaluation 기준 만점)
- 평균 점수와 편차 표시
- 점수가 낮은 코드가 있으면 어떤 항목이 약한지 표시
```

### 검증하기

1. 2단계에서 개선한 **최종 프롬프트를 그대로 두고**
2. 앱에서 5개 코드 검증 결과를 확인

### 관찰하기

- **점수가 비슷하면**: 프롬프트가 범용적으로 잘 작동하는 것
- **특정 코드에서 확 떨어지면**: 프롬프트가 sample-code-1에만 맞춰진 것
- 떨어진 항목을 보고 프롬프트를 보강 → 다시 돌려보기

> **이것이 autoresearch에서 validation set의 역할입니다.**
> 하나의 코드에만 잘 맞는 프롬프트가 아니라, 어떤 코드에도 통하는 프롬프트를 만드는 것.
