```
## 화면 구성

### 입력 영역
- "프롬프트" 텍스트 영역 (코드 리뷰 지침서를 입력)

### 결과 영역

  - AI가 생성한 코드 리뷰 결과물
  - 10개 항목별 점수표 (이전 대비 +/- 표시)

## 동작 방식

"자동 개선 시작"을 누르면 내장된 Gemini 2.5 Flash 모델이 자동으로:
1. 프롬프트를 사용해서 테스트 코드에 대한 코드 리뷰 생성
2. 평가 기준으로 코드 리뷰를 채점
3. 점수가 낮은 항목 분석 → 프롬프트 개선
4. 개선된 프롬프트로 1번부터 반복
매 사이클 완료 시 바 차트 업데이트.
```

## 2단계: GitHub 코드 연동으로 실제 검증

<Timer minutes={20} />

<Callout type="default">
**이 단계의 목표**: 1단계에서 개선된 프롬프트가 "다른 코드에도 통하는지" 검증하기

시험공부에 비유하면 이렇습니다:
- 1단계는 <strong>연습문제로 공부한 것</strong>입니다 — 같은 코드로 개선하고, 같은 코드로 채점했으니 점수가 잘 나옵니다.
- 2단계는 <strong>처음 보는 시험 문제로 실력을 확인하는 것</strong>입니다 — 다른 코드로 테스트해서 진짜 실력을 봅니다.

autoresearch에서도 같은 원리입니다. AI가 코드를 개선할 때 쓴 데이터가 아니라,
<strong>한 번도 본 적 없는 별도의 데이터</strong>(`val_bpb`)로 측정합니다.
연습문제만 잘 푸는 건지, 진짜 실력이 있는 건지 구분하기 위해서입니다.
</Callout>

### 저장소 fork + Gemini 새 탭에서 연결

```
## 화면 구성

### 1. 소스 코드 미리보기 영역
- 저장소에서 가져온 5개 코드를 탭으로 보여주기
  (출금 | 자동매수 | 화이트리스트 | 거래내역 | 알림)
- 각 탭에 해당 코드의 내용과 "참고 맥락"을 표시
- 코드가 정상적으로 로드되었음을 확인할 수 있도록

### 2. 프롬프트 입력 영역
- "코드 리뷰 프롬프트" 텍스트 영역
- "검증 시작" 버튼

### 3. 진행 상황 영역
- "검증 시작"을 누르면 5개 코드를 순서대로 분석
- 각 코드 분석 시 아래 단계를 스텝 바이 스텝으로 보여주기:
  - Step 1/3: "출금 코드 리뷰 생성 중..." → 완료 시 ✅
  - Step 2/3: "평가 기준으로 채점 중..." → 완료 시 ✅ + 점수 표시
  - Step 3/3: "다음 코드로 이동..." → 완료 시 ✅
- 현재 진행 중인 코드가 몇 번째인지 프로그레스 표시 (예: 2/5)
- 전체 진행률 프로그레스 바

### 4. 결과 영역 (모든 코드 분석 완료 후)

#### 4-1. 총점 비교 바 차트
- 5개 코드별 총점 바 차트 (가로 배치, 50점 만점)
- 코드 이름: 출금 | 자동매수 | 화이트리스트 | 거래내역 | 알림
- "평균 점수"와 "편차" 표시 — 편차가 크면 프롬프트가 범용적이지 않다는 뜻

#### 4-2. 항목별 점수 히트맵
- 10행(평가 항목) x 5열(코드) 테이블
- 각 셀에 점수 + 배경색
- 행 끝에 해당 항목의 평균 점수 표시
- 가장 낮은 항목이 무엇인지 한눈에 파악 가능

#### 4-3. 코드별 상세 평가 (아코디언)
각 코드를 클릭하면 펼쳐지는 상세 영역:
- AI가 생성한 코드 리뷰 전문
- 10개 항목별 점수표:
  | 항목 | 점수 | 채점 근거 |
  - 채점 근거: 해당 점수를 준 구체적 이유 (예: "보안 위험 2건 지적했으나 코드 예시 없음 → 3점")
- 발견한 이슈 목록 (심각도별 분류)
- 놓친 이슈 목록 (프롬프트에 없어서 못 찾은 것)

#### 4-4. 종합 분석
- 프롬프트의 강점: 점수가 일관되게 높은 항목
- 프롬프트의 약점: 점수가 낮거나 편차가 큰 항목
- 개선 제안: "이 항목의 점수를 올리려면 프롬프트에 이런 내용을 추가하세요"
- 최종 프롬프트 복사 버튼

## 디자인
- 한국어 UI
- 다크 모드
- 바 차트 색상: 15점 이하 회색, 16~30 파란색, 31 이상 초록색
- 히트맵: 1~2점 빨간, 3점 노란, 4~5점 초록
- 진행 중 단계는 파란 하이라이트, 완료 단계는 초록 체크
```

</Tabs.Tab>
</Tabs>

### 검증하기

1. 1단계에서 만든 최종 프롬프트를 앱에 붙여넣기
2. **"검증 시작"** 클릭
3. 5개 코드에 대한 점수 비교를 확인

### 관찰하기

- **바 차트가 고르면**: 프롬프트가 범용적으로 잘 작동하는 것
- **특정 코드만 점수가 낮으면**: 프롬프트가 특정 패턴에만 맞춰진 것
- **히트맵에서 빨간 셀이 많은 항목**: 프롬프트에서 해당 관점이 약하다는 뜻
- **편차가 크면**: 프롬프트를 보강한 뒤 다시 검증

> **이것이 autoresearch에서 validation set의 역할입니다.**
> 하나의 코드에만 잘 맞는 프롬프트가 아니라, 어떤 코드에도 통하는 프롬프트를 만드는 것.

## 3단계: 내 업무 프롬프트로 실험

<Timer minutes={20} />

<Callout type="default">
**이 단계의 목표**: 당장 내일부터 바로 쓸 수 있는, 내 업무에 맞는 프롬프트 만들기
</Callout>

### 내 업무 샘플 준비하기

코드 리뷰가 아닌 <strong>본인의 실제 반복 업무</strong>로 바꿔봅니다.
본인이 AI에게 반복적으로 시키고 싶은 업무의 **입력물(코드, 문서, 기획서 등)을 5개** 준비합니다.

**방법 A**: 실제 업무 자료가 있다면 5개를 가져옵니다.

| 직군 | 가져올 수 있는 것 예시 |
|------|---------------------|
| BE 개발 | 최근 PR의 코드 5개, API 스펙 문서 5개 |
| FE 개발 | 최근 작성한 컴포넌트 코드 5개, CSS 파일 5개 |
| PM | 최근 작성한 기획서 5개, 요구사항 문서 5개 |
| QA | 최근 테스트한 기능 설명서 5개, 결함 리포트 5개 |

**방법 B**: 지금 당장 가져오기 어렵다면, Gemini에게 내 업무와 비슷한 샘플 5개를 만들게 합니다.

```
나는 [직군]이고, [구체적인 업무 설명]을 반복적으로 하고 있어.

예를 들어 최근에 이런 일을 했어:
- [실제 업무 예시 1]
- [실제 업무 예시 2]

이 업무를 AI에게 시키기 위한 프롬프트를 테스트하려고 해.
테스트에 쓸 수 있는 현실적인 샘플을 5개 만들어줘.

조건:
- 내 업무 맥락과 비슷한 난이도와 복잡도로
- 각 샘플마다 서로 다른 유형의 문제점을 포함해줘
- 겉보기에는 괜찮아 보이지만 자세히 보면 개선할 수 있는 문제가 있는 형태로
- 너무 명백한 실수가 아니라, 도메인 지식이 있어야 발견할 수 있는 문제로
```

### 자동 개선 돌리기

1. 내 업무에 맞는 프롬프트를 작성 (예: "이 기획서의 예외 케이스를 도출해줘")
2. 준비한 샘플 5개 + 내 프롬프트로 1단계 Canvas 앱에서 자동 개선 사이클 실행
3. 결과를 보고 방향 조정 → 재실행
4. 시간이 남으면 2단계 검증 앱에 5개 샘플을 넣고 범용성 검증

### 최종 프롬프트 저장

- 개선된 프롬프트를 메모장이나 GitHub fork에 저장
- 이것이 <strong>당장 내일부터 바로 쓸 수 있는 업무 프롬프트</strong>입니다
