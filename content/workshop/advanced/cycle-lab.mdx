# 실습: AI가 코드를 자동 개선하게 만들기

<Timer minutes={30} />

<Callout type="warning">
**autoresearch에서 AI가 개선하는 것은 코드(train.py)이지, 지시서(program.md)가 아닙니다.**

사람은 program.md(지시서)를 다듬고, AI는 그 지시를 따라 코드를 반복적으로 개선합니다.
오늘 실습도 같은 구조입니다: 사람은 프롬프트(지시서)를 다듬고, AI는 그 프롬프트를 따라 <strong>코드를 반복적으로 개선</strong>합니다.
</Callout>

| autoresearch | 오늘 실습 |
|---|---|
| `program.md` — 사람이 다듬는 지시서 | 코드 리뷰 프롬프트 — 사람이 다듬는 지시서 |
| `train.py` — AI가 개선하는 코드 | 샘플 코드 — AI가 리뷰하고 개선하는 대상 |
| `val_bpb` — 코드 실행 결과로 측정 | evaluation 기준 — 코드 품질을 측정 |

3단계로 점점 깊어집니다. 각자 속도에 맞춰 진행하세요.

| 단계 | 하는 일 | 왜 하는지 |
|------|---------|----------|
| **1단계** | 저장소 연결 → 프롬프트로 코드 리뷰 + 채점 | "내 프롬프트로 AI가 코드를 얼마나 잘 개선하는지" 베이스라인 |
| **2단계** | 자동 개선 Canvas 앱 만들기 → 사이클 돌리기 | AI가 리뷰→개선→채점을 자동 반복 |
| **3단계** | 같은 프롬프트로 다른 코드도 개선해보기 | "이 지시서가 범용적인지" 확인 → 프롬프트 다듬기 |

## 1단계: 저장소 연결 + 수동 1사이클

<Callout type="default">
**이 단계의 목표**: 내 프롬프트(지시서)로 AI가 코드를 얼마나 잘 리뷰하는지 베이스라인 확인

autoresearch에서도 첫 실행은 항상 train.py를 있는 그대로 돌려서 베이스라인을 잡습니다.
</Callout>

3. fork한 저장소 URL을 입력하고 **가져오기** 클릭

![코드 가져오기 — prompt-workshop 저장소 URL 입력](/workshop/advanced/gemini-import-prompt-workshop.png)

### 수동으로 한 번 돌려보기

```
이 저장소의 구조를 먼저 파악해줘.

prompts/code-review.md를 지시서로 사용해서
samples/sample-code-1.md의 코드를 리뷰하고,
리뷰 결과를 반영해서 개선된 코드를 만들어줘.

그리고 개선된 코드를 evaluation/code-review.md 기준으로 채점해줘.
원본 코드 대비 어떤 점이 개선되었는지도 알려줘.
```

### 관찰하기

| 유형 | 예상 결과 | 이유 |
|------|----------|------|
| **AI가 원래 잘 찾는 문제** | 개선됨 | 프롬프트가 단순해도 AI가 기본적으로 찾아서 고침 |
| **프롬프트에 지침이 있어야 찾는 문제** | 그대로 남음 | 프롬프트에 해당 지침이 없으면 AI가 문제로 인식하지 않음 |

**개선되지 않고 남아있는 문제가 많을수록** — 프롬프트를 더 잘 써야 한다는 뜻입니다.

<Callout type="info">
"프롬프트를 좀 더 구체적으로 쓰면 AI가 이 문제도 잡을 텐데..."
— 바로 그걸 AI가 자동으로 반복해주는 앱을 다음 단계에서 만듭니다.
</Callout>

</Callout>

## 저장소에서 가져올 파일

평가 기준:
- evaluation/code-review.md의 내용을 그대로 따를 것
  (항목 수, 점수 범위, 채점 근거 모두 이 파일 기준)

개선할 코드:
- samples/sample-code-1.md (이 코드를 반복적으로 개선)

## 화면 구성

### 개선 대상 코드 미리보기
- 저장소에서 가져온 samples/sample-code-1.md의 코드를 표시
- "참고 맥락"도 함께 표시
- 이 코드가 사이클을 거치며 개선되는 것을 추적

### 입력 영역
- "프롬프트 (지시서)" 텍스트 영역 — AI가 코드를 리뷰할 때 따를 지침
- "사이클 수" 선택 (1~5, 기본값 3)
- "자동 개선 시작" 버튼

### 결과 영역

#### 상단: 사이클별 코드 품질 점수 차트
- v0(원본)→v1→v2... 코드 품질 점수 변화를 시각화
- keep된 사이클은 초록, discard된 사이클은 빨간색으로 구분
- 점수는 단조 증가해야 함 (discard 시 이전 점수 유지)

#### 하단: 사이클별 상세 (탭 또는 아코디언)
각 사이클을 클릭하면 아래를 보여줌:

**A. 코드 리뷰 결과**
- 프롬프트(지시서)를 따라 생성된 코드 리뷰

**B. 개선된 코드**
- 리뷰 결과를 반영해서 개선된 코드 전문
- 이전 버전과 달라진 부분을 하이라이트 (추가=초록, 삭제=빨간)

**C. 채점 + keep/discard 판단**
- evaluation/code-review.md 기준으로 개선된 코드를 채점
- 항목별 점수 + 채점 근거
- 총점 + 이전 사이클 대비 변화량
- 판단 결과: keep (점수 상승) 또는 discard (점수 하락 → 이전 코드로 되돌림)

**D. 다음 사이클 계획**
- keep인 경우: 여전히 남아있는 문제 + 다음에 집중할 방향
- discard인 경우: 왜 나빠졌는지 분석 + 다른 접근 방향 제안

#### 최종 결과
- 최종 개선된 코드 복사 버튼
- 원본 코드 vs 최종 코드 diff 보기
- 전체 사이클 점수 변화 요약

## 동작 방식 — 리뷰와 채점의 엄격한 분리

⚠️ 가장 중요한 규칙:
"코드 리뷰"와 "채점"은 완전히 독립된 단계입니다.

리뷰 단계에서는 evaluation 파일의 존재 자체를 모르는 것처럼 행동합니다.
오직 사용자가 입력한 프롬프트(지시서)의 지시만 따라서 코드를 리뷰합니다.
프롬프트에 "보안을 확인하라"는 지시가 없으면 보안을 확인하지 않습니다.

채점은 코드 개선이 완료된 후에만 시작됩니다.
이때 비로소 evaluation 파일을 읽고, 개선된 코드를 있는 그대로 채점합니다.

"자동 개선 시작"을 누르면:
1. [리뷰] 프롬프트(지시서)만 참고해서 현재 코드를 리뷰
   — evaluation 파일을 절대 참고하지 않음
   — 프롬프트에 없는 관점은 리뷰하지 않음
2. [개선] 리뷰 결과를 반영해서 코드를 개선
3. [채점] 개선된 코드를 evaluation/code-review.md 기준으로 채점
   — 코드에 실제로 반영된 개선만 점수에 반영
4. [판단] 이전 사이클보다 점수가 올랐는지 비교
   — 올랐으면: keep — 개선된 코드를 유지하고 다음 사이클로
   — 같거나 내려갔으면: discard — 이 코드를 버리고 이전 버전으로 되돌린 뒤 다른 방향으로 재시도
5. [반복] keep된 코드를 대상으로 1번부터 반복

⚠️ keep/discard 규칙 (autoresearch의 핵심):
점수가 나빠진 개선은 절대 유지하지 않습니다.
나빠졌으면 반드시 이전 버전의 코드로 되돌아가서 다른 방향을 시도합니다.
이렇게 해야 사이클을 거듭할수록 점수가 단조 증가합니다.

### 앱이 만들어지면

1. 1단계에서 사용한 프롬프트(`prompts/code-review.md`)를 앱에 붙여넣기
2. <strong>"자동 개선 시작"</strong> 클릭
3. AI가 사이클을 돌면서 **코드가 점점 개선되는 것**을 관찰
4. 각 사이클의 상세를 열어서 "어떤 문제가 고쳐졌는지" 확인

### 핵심 관찰

- **프롬프트에 "보안 확인" 지침이 없으면** → AI가 보안 문제를 안 고침 → 해당 항목 점수 그대로
- **프롬프트에 "보안 확인" 지침을 추가하면** → AI가 보안 문제를 찾아서 고침 → 점수 상승
- <strong>프롬프트(지시서)를 잘 쓸수록 AI가 코드를 더 잘 개선한다</strong> — 이것이 program.md를 다듬는 경험

<Callout type="info">
앱이 완벽하지 않아도 됩니다.
핵심은 <strong>"프롬프트(지시서)의 품질이 코드 개선의 품질을 결정한다"</strong>는 것을 직접 확인하는 것입니다.
</Callout>

## 3단계: 같은 프롬프트로 다른 코드도 개선되는가?

<Callout type="default">
**이 단계의 목표**: 2단계에서 사용한 프롬프트(지시서)가 다른 코드에서도 잘 작동하는지 확인하기

2단계에서 프롬프트(지시서)는 고정한 채 AI가 sample-code-1을 잘 개선했습니다.
하지만 이 프롬프트가 <strong>sample-code-1에만 잘 맞는 건지, 어떤 코드에나 통하는 건지</strong>는 아직 모릅니다.

같은 프롬프트로 다른 코드도 돌려봐서, 프롬프트의 범용성을 확인합니다.
</Callout>

### 앱에 검증용 코드 추가하기

2단계에서 만든 앱에 샘플 코드를 추가합니다.
Gemini 대화에서 아래와 같이 요청하세요:

```
이 앱에 검증 기능을 추가해줘.

저장소의 나머지 4개 샘플 코드를 추가로 가져와서,
같은 프롬프트(지시서)로 각 코드를 리뷰 + 개선 + 채점한 결과를
비교할 수 있게 해줘.

추가할 코드:
- samples/sample-code-2.md (자동 매수 예약)
- samples/sample-code-3.md (주소 화이트리스트)
- samples/sample-code-4.md (거래 내역 조회/CSV)
- samples/sample-code-5.md (알림 설정/발송)

검증 결과 표시:
- 5개 코드별 개선 후 총점 바 차트 (가로 배치)
- 평균 점수와 편차 표시
- 점수가 낮은 코드가 있으면 어떤 항목이 약한지 표시
```

### 관찰하기

- **5개 코드 모두 잘 개선되면**: 이 프롬프트(지시서)는 범용적으로 잘 작동하는 것
- **특정 코드에서 개선이 부족하면**: 프롬프트에 해당 코드 유형에 대한 지침이 빠져있다는 뜻
- 부족한 부분을 프롬프트에 추가 → 다시 돌려보기 — **이것이 program.md를 다듬는 경험**

> 2단계에서는 AI가 코드를 개선했고, 3단계에서는 사람이 프롬프트를 다듬습니다.
> 두 단계를 오가면서 "더 좋은 지시서"를 만들어가는 것이 autoresearch의 전체 사이클입니다.
