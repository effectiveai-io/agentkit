# 실습: 자동 개선 사이클

3단계로 점점 깊어집니다.

| 단계 | 핵심 | 결과 |
|------|------|------|
| **1단계** | Canvas 앱으로 자동 개선이 뭔지 체감 | "AI가 프롬프트를 알아서 고치네" |

## 화면 구성

### 입력 영역
- "프롬프트" 텍스트 영역 (코드 리뷰 지침서를 입력)
- "테스트 코드" 텍스트 영역 (리뷰할 코드를 입력)
- "평가 기준" 텍스트 영역 (10개 항목, 각 1~5점, 만점 50점)
- "사이클 수" 선택 (1~5, 기본값 3)
- "자동 개선 시작" 버튼

### 결과 영역
- 사이클별 총점 바 차트 (50점 만점, v0→v1→v2... 점수 변화 시각화)
- 각 사이클 상세 (아코디언으로 접기/펴기):
  - 해당 사이클에서 사용한 프롬프트
  - AI가 생성한 코드 리뷰 결과물
  - 10개 항목별 점수표 (이전 대비 +/- 표시)
  - 이번에 개선한 것 / 다음에 개선할 것
- 최종 프롬프트 복사 버튼

## 동작 방식

"자동 개선 시작"을 누르면 내장된 Gemini 2.5 Flash 모델이 자동으로:
1. 프롬프트를 사용해서 테스트 코드에 대한 코드 리뷰 생성
2. 평가 기준으로 코드 리뷰를 채점
3. 점수가 낮은 항목 분석 → 프롬프트 개선
4. 개선된 프롬프트로 1번부터 반복
매 사이클 완료 시 바 차트 업데이트.

## 평가 기준 기본값

"10개 항목, 각 1~5점 (만점 50점)
1. 네이밍 컨벤션 — 팀 규칙 기반 지적 여부

3. 보안 취약점 — 도메인 특화 보안 이슈 지적
4. 금액/데이터 정확성 — 타입, 정밀도 이슈
5. 동시성/경쟁 조건 — 동시 요청 시 위험 시나리오
6. 에러 처리 — 누락된 예외 케이스
7. 테스트 용이성 — 테스트 가능한 구조 제안
8. 리뷰 심각도 분류 — MUST/SHOULD/NICE 구분
9. 코드 예시 품질 — Before/After 코드 포함 여부
10. 도메인 우선순위 — 보안>금액>성능 순 리뷰

1점: 언급 없음
2점: 언급만
3점: 일부 반영

## 2단계: GitHub 코드 연동으로 실제 검증

<Timer minutes={20} />

<Callout type="default">
**이 단계의 목표**: 1단계에서 개선된 프롬프트가 "다른 코드에도 통하는지" 검증하기

시험공부에 비유하면 이렇습니다:
- 1단계는 <strong>연습문제로 공부한 것</strong>입니다 — 같은 코드로 개선하고, 같은 코드로 채점했으니 점수가 잘 나옵니다.
- 2단계는 <strong>처음 보는 시험 문제로 실력을 확인하는 것</strong>입니다 — 다른 코드로 테스트해서 진짜 실력을 봅니다.

autoresearch에서도 같은 원리입니다. AI가 코드를 개선할 때 쓴 데이터가 아니라,
<strong>한 번도 본 적 없는 별도의 데이터</strong>(`val_bpb`)로 측정합니다.
연습문제만 잘 푸는 건지, 진짜 실력이 있는 건지 구분하기 위해서입니다.
</Callout>

### 저장소 fork + Gemini 새 탭에서 연결

1. [prompt-workshop](https://github.com/effectiveai-io/prompt-workshop) 저장소를 fork
2. **Gemini 새 탭**을 열고 새 대화를 시작
3. fork한 저장소를 연결 (+ 버튼 → 코드 가져오기 → fork한 저장소 URL)
4. **Canvas 모드**를 켜기

### Canvas로 검증 앱 만들기

1단계에서 만든 최종 프롬프트를 복사해둔 뒤, Canvas에서 검증 앱을 만듭니다:

<Tabs.Tab>

```
이 저장소를 참고해서, Gemini Canvas로 "프롬프트 검증기" 웹앱을 만들어줘.

이 앱은 하나의 코드 리뷰 프롬프트가 여러 코드에서 얼마나 잘 작동하는지 검증하는 앱이야.
내장된 Gemini 2.5 Flash 모델을 사용해줘.

## 동작 방식

1. 사용자가 "코드 리뷰 프롬프트"를 입력
2. "검증 시작" 버튼을 누르면, 저장소의 samples/ 폴더에 있는 5개 코드 파일 각각에 대해:
   - 입력된 프롬프트로 코드 리뷰 생성
   - evaluation/code-review.md 기준으로 채점 (10항목, 각 1~5점, 만점 50점)
3. 5개 코드의 점수를 비교 테이블 + 바 차트로 시각화

## 화면 구성

### 입력 영역
- "코드 리뷰 프롬프트" 텍스트 영역
- "검증 시작" 버튼

### 결과 영역
- 5개 코드별 총점 바 차트 (가로 배치, 50점 만점)
  - 코드 이름: 출금 | 자동매수 | 화이트리스트 | 거래내역 | 알림
- 10개 항목별 점수 비교 히트맵 또는 테이블
  - 행: 10개 평가 항목
  - 열: 5개 코드
  - 점수가 낮은 셀은 빨간색, 높은 셀은 초록색
- 각 코드별 상세 리뷰 (아코디언으로 접기/펴기)
- "평균 점수"와 "편차" 표시 — 편차가 크면 프롬프트가 범용적이지 않다는 뜻
- 최종 프롬프트 복사 버튼

## 검증할 코드 파일
저장소의 아래 5개 파일을 사용:
- samples/sample-code-1.md (출금 요청 처리)
- samples/sample-code-2.md (자동 매수 예약)
- samples/sample-code-3.md (주소 화이트리스트)
- samples/sample-code-4.md (거래 내역 조회/CSV)
- samples/sample-code-5.md (알림 설정/발송)

## 디자인
- 한국어 UI
- 다크 모드
- 바 차트 색상: 15점 이하 회색, 16~30 파란색, 31 이상 초록색
- 히트맵: 1~2점 빨간, 3점 노란, 4~5점 초록
```

</Tabs.Tab>
</Tabs>

### 검증하기

1. 1단계에서 만든 최종 프롬프트를 앱에 붙여넣기
2. **"검증 시작"** 클릭
3. 5개 코드에 대한 점수 비교를 확인

### 관찰하기

- **바 차트가 고르면**: 프롬프트가 범용적으로 잘 작동하는 것
- **특정 코드만 점수가 낮으면**: 프롬프트가 특정 패턴에만 맞춰진 것
- **히트맵에서 빨간 셀이 많은 항목**: 프롬프트에서 해당 관점이 약하다는 뜻
- **편차가 크면**: 프롬프트를 보강한 뒤 다시 검증

> **이것이 autoresearch에서 validation set의 역할입니다.**
> 하나의 코드에만 잘 맞는 프롬프트가 아니라, 어떤 코드에도 통하는 프롬프트를 만드는 것.

## 3단계: 내 업무 프롬프트로 실험

<Timer minutes={20} />

<Callout type="default">
**이 단계의 목표**: 당장 내일부터 바로 쓸 수 있는, 내 업무에 맞는 프롬프트 만들기
</Callout>

### 내 프롬프트 입력

코드 리뷰가 아닌 **본인의 실제 반복 업무**로 바꿔봅니다.

| 직군 | 프롬프트 예시 |
|------|-------------|
| BE 개발 | "PR 설명 자동 작성", "API 스펙 리뷰" |
| FE 개발 | "컴포넌트 접근성 리뷰", "CSS 리팩토링 제안" |
| PM | "기획서 예외 케이스 도출", "요구사항 명확화" |
| QA | "TC 설계", "결함 리포트 작성" |

### Gemini로 샘플 만들기

실제 업무 코드/문서가 없으니, Gemini에게 샘플을 만들게 합니다:

```
나는 [직군]이고, [업무 설명]을 반복적으로 하고 있어.
이 업무를 AI에게 시키기 위한 프롬프트를 테스트하고 싶은데,
테스트에 쓸 수 있는 현실적인 샘플을 3개 만들어줘.
각 샘플에는 일부러 개선할 수 있는 문제점을 포함해줘.
```

### 자동 개선 돌리기

1. 내 프롬프트 + Gemini가 만든 샘플 3개로 자동 개선 사이클 실행
2. 또는 1단계에서 만든 Canvas 앱에 내 프롬프트와 샘플을 넣고 돌리기
3. 결과를 보고 방향 조정 → 재실행

### 최종 프롬프트 저장

- 개선된 프롬프트를 메모장이나 GitHub fork에 저장
- 이것이 <strong>당장 내일부터 바로 쓸 수 있는 업무 프롬프트</strong>입니다

<Callout type="info">
돌아가서 할 것: 실제 업무 코드/문서를 샘플로 넣고 다시 돌리면,
지금보다 훨씬 정교한 프롬프트를 얻을 수 있습니다.
</Callout>
