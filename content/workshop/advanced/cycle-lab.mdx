# 실습: fork + 자동 개선 사이클

<Steps>

## Step 1: 점수 추적 앱 만들기

<Timer minutes={5} />

자동 개선 사이클을 돌리기 전에, **매 사이클의 점수를 기록하고 시각화하는 앱**을 Gemini Canvas로 먼저 만듭니다.
이 앱에 매 사이클 결과를 입력하면서, 프롬프트가 점점 좋아지는 과정을 눈으로 확인합니다.

Gemini에서 Canvas 모드를 켜고 아래 프롬프트를 입력하세요:

<Tabs items={['점수 추적 앱 프롬프트']}>
<Tabs.Tab>

```
Gemini Canvas로 "프롬프트 개선 스코어 트래커" 웹앱을 만들어줘.

기능:
- 5개 평가 항목별 점수(0, 1, 2)를 입력할 수 있는 폼
- "사이클 추가" 버튼을 누르면 해당 사이클의 점수가 기록됨
- 기록된 사이클들의 총점을 바 차트로 시각화 (v0, v1, v2... 순서로)
- 항목별 점수 변화도 테이블로 보여주기 (이전 대비 +1, -1 표시)
- 차트 색상: 총점이 낮으면 회색, 중간이면 파란색, 높으면 초록색

평가 항목 (기본값, 사용자가 수정 가능):
1. 팀 컨벤션 반영
2. 보안/성능 관점
3. 비즈니스 맥락 이해
4. 구체적 개선 제안
5. 심각도 분류

디자인:
- 한국어 UI
- 다크 모드
- 깔끔하고 컴팩트한 레이아웃
- 상단에 바 차트, 하단에 입력 폼
```

</Tabs.Tab>
</Tabs>

<Callout type="info">
이 앱을 브라우저 탭에 띄워두세요. 이후 매 사이클마다 여기에 점수를 기록합니다.
평가 항목은 본인 직군에 맞게 수정해도 좋습니다.
</Callout>

## Step 2: fork + 본인 맥락 수정

<Timer minutes={15} />

### 저장소 fork

1. [prompt-workshop](https://github.com/effectiveai-io/prompt-workshop) 접속
2. 우측 상단 **Fork** 클릭 → 본인 GitHub 계정으로 fork

### 본인 직군에 맞는 파일 선택

| 직군 | 파일 |
|------|------|
| BE/FE 개발 | `prompts/code-review.md` |
| QA | `prompts/test-case.md` |
| PM/기획 | `prompts/spec-review.md` |
| 공통 | `prompts/document-draft.md` |

### GitHub 웹에서 수정

1. fork한 저장소에서 선택한 파일 클릭
2. 연필 아이콘(Edit) 클릭
3. 본인 팀 맥락에 맞게 수정:
   - 팀명, 서비스명을 본인 것으로
   - "팀 컨벤션" 섹션에 실제 규칙 추가
   - "도메인 맥락" 섹션에 우리 서비스 특수 규칙 추가
4. Commit changes

<Callout type="info">
지금 실제 팀 코드가 없으니 완벽하지 않아도 됩니다.
돌아가서 실제 맥락을 추가하면 진짜 자산이 됩니다.
</Callout>

## Step 3: Gemini에 연결 + 수동 1사이클

<Timer minutes={10} />

### 저장소 연결

Gemini 웹에서 **새 대화**를 시작하고, fork한 저장소를 연결합니다.

### 수동 사이클 (감 잡기 + 베이스라인)

```
이 저장소의 구조를 먼저 파악해줘.
그리고 prompts/code-review.md를 사용해서
samples/sample-code.md의 코드를 리뷰해줘.
리뷰한 결과를 evaluation/code-review.md 기준으로 자체 채점하고,
아래 형식으로 점수표를 보여줘:

| 항목 | 점수 (0~2) | 이유 |
|------|-----------|------|
| 팀 컨벤션 반영 | ?/2 | ... |
| 보안/성능 관점 | ?/2 | ... |
| 비즈니스 맥락 이해 | ?/2 | ... |
| 구체적 개선 제안 | ?/2 | ... |
| 심각도 분류 | ?/2 | ... |
| **총점** | **?/10** | |
```

<Callout type="warning">
**점수표가 나오면 → 점수 추적 앱에 v0 점수를 입력하세요.**

이 점수가 베이스라인입니다. 여기서부터 얼마나 올라가는지 추적합니다.
</Callout>

**관찰 포인트:**
- 어떤 항목이 0점인가? → 프롬프트에 해당 내용이 아예 없다는 뜻
- 어떤 항목이 1점인가? → 언급은 있지만 구체성이 부족하다는 뜻

## Step 4: 자동 개선 사이클 실행

<Timer minutes={20} />

<Callout type="warning">
**여기가 autoresearch 본질을 경험하는 순간입니다.**

사람이 직접 고치는 게 아니라, AI가 스스로 개선 루프를 돌게 시킵니다.
매 사이클마다 점수표가 나오면 점수 추적 앱에 기록하세요.
</Callout>

### 메타 프롬프트

```
이 저장소의 program.md를 읽고 지시대로 따라해줘.
prompts/code-review.md를 개선 대상으로,
evaluation/code-review.md를 평가 기준으로,
samples/sample-code.md를 테스트 시나리오로 사용해서
자동 개선 사이클을 3회 실행해줘.

중요: 매 사이클이 끝날 때마다 반드시 아래 형식의 점수표를 출력해줘.

## Cycle N 결과

| 항목 | 점수 (0~2) | 이전 대비 |
|------|-----------|----------|
| 팀 컨벤션 반영 | ?/2 | +? |
| 보안/성능 관점 | ?/2 | +? |
| 비즈니스 맥락 이해 | ?/2 | +? |
| 구체적 개선 제안 | ?/2 | +? |
| 심각도 분류 | ?/2 | +? |
| **총점** | **?/10** | **+?** |

이번에 개선한 것: (한 줄)
다음에 개선할 것: (한 줄)
```

→ Gemini가 `program.md`의 지시를 따라 자동으로:
1. 현재 프롬프트로 결과물 생성
2. 평가 기준으로 자체 채점 → **점수표 출력**
3. 점수가 낮은 항목 분석 → 프롬프트 개선
4. 개선된 프롬프트로 다시 반복

<Callout type="info">
**매 사이클의 점수표가 나올 때마다 점수 추적 앱에 입력하세요.**

바 차트가 올라가는 것을 실시간으로 확인할 수 있습니다.
</Callout>

### 관찰 + 방향 조정

점수 추적 앱의 차트를 보면서 관찰합니다:
- 어떤 항목의 점수가 올라가고 있는지
- 어떤 항목이 계속 낮은 채로인지

특정 항목이 계속 낮으면 <strong>추가 지시로 방향을 조정</strong>합니다:

```
보안/성능 관점 점수가 계속 1점인데,
다음 사이클에서는 이 항목을 우선적으로 강화해줘.
특히 금융 서비스에서 민감정보 로깅 관련 규칙을 추가해줘.
```

> **점수 추적 앱에서 낮은 항목을 보고 → Gemini에게 방향을 조정하는 것.**
> **이것이 `program.md`를 프로그래밍하는 경험입니다.**

## Step 5: 결과 확인 + 저장

<Timer minutes={10} />

### 점수 추적 앱에서 전체 흐름 확인

- v0 → v1 → v2 → v3 총점 변화를 확인
- **가장 큰 점수 상승이 있었던 사이클**: 그때 뭘 바꿨는지가 핵심 인사이트
- **여전히 낮은 항목**: 돌아가서 실제 맥락을 추가하면 올릴 수 있는 부분

### 최종 프롬프트 저장

AI가 만든 최종 버전을 확인합니다:
- 마음에 드는 개선 사항 체크
- 불필요한 변경 제거
- 최종 프롬프트를 GitHub에 커밋하거나 메모장에 저장

</Steps>

<Callout type="info">
autoresearch는 밤새 100번 돌립니다. 여러분은 3~4번 돌렸는데도 차트가 올라갔죠.
돌아가서 실제 코드, 실제 기획서를 `samples/`에 넣고 다시 돌리면 어떻게 될까요?
</Callout>
