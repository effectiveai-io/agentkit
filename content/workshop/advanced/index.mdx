# 워크숍 심화 — 지침서를 프로그래밍하라

<div className="workshop-narrative">

> **지침서 작성 → AI가 업무 결과물 생성 → 평가 → 지침서 개선 → 반복**

이 워크숍에서는 Andrej Karpathy의 <strong>autoresearch</strong> 프로젝트에서 영감을 받은 루프를 적용합니다.
코드를 짜는 대신 <strong>AI에게 줄 업무 지침서를 3사이클 반복 개선</strong>하고,
매 사이클마다 점수가 오르는 것을 직접 체감합니다.

</div>

## 3시간, 핵심 목표

<Callout type="default">
**"월요일 아침에 바로 쓸 수 있는 업무 지침서를 만들어 간다."**

도구 체험이 아닙니다. 내 반복 업무에 대한 AI 지침서를 만들고,
3번의 개선 사이클을 거쳐 실제로 품질이 올라가는 것을 측정합니다.
</Callout>

구체적으로:
- 내 반복 업무의 <strong>평가 기준</strong>을 정의한다
- 지침서를 3사이클 돌려 <strong>개선 곡선</strong>을 체감한다
- 교환 테스트로 <strong>"남이 써도 작동하는가?"</strong>를 검증한다
- 우아한형제들 실제 사례와 연결해 <strong>팀 자산화 경로</strong>를 그린다

## 핵심 경험

> **"0점 → 4점 → 8점" — 이 곡선을 직접 경험하는 것이 이 워크숍의 전부입니다.**

Cycle 1에서 빈 지침서로 AI에게 시키면, 일반적이고 뻔한 결과가 나옵니다.
Cycle 2에서 팀 컨벤션을 넣으면, 확실히 구체적인 결과가 나옵니다.
Cycle 3에서 도메인 맥락까지 넣으면, 전문가 수준의 결과가 나옵니다.

이 경험이 곧 <strong>"지침서를 프로그래밍한다"</strong>는 것의 의미입니다.

## 4세션 흐름

<Steps>
### autoresearch 루프 이해하기

<span className="step-tools">15분</span>

Karpathy의 autoresearch가 어떻게 작동하는지,
그리고 이것을 우리 업무에 어떻게 적용하는지 이해합니다.
우아한형제들의 테스트 자동화 사례로 "AI에게 구조를 줘야 하는 이유"를 확인합니다.

[autoresearch 루프 →](/workshop/advanced/autoresearch)

### 지침서 개선 사이클 3회전

<span className="step-tools">75분 (준비 25분 + 사이클 50분)</span>

내 반복 업무를 하나 선택하고, 평가 기준을 정의합니다.
그리고 지침서 v0 → v1 → v2, 3사이클을 돌리며 매번 점수를 기록합니다.

[지침서 개선 사이클 →](/workshop/advanced/cycle)

### 교환 테스트

<span className="step-tools">30분</span>

짝과 지침서를 교환해서 "내가 아닌 사람이 써도 작동하는가?"를 테스트합니다.
개인 실험에서 팀 자산으로 넘어가는 경계선이 여기입니다.

[교환 테스트 →](/workshop/advanced/exchange)

### 반복에서 축적으로

<span className="step-tools">50분 (사례 25분 + 회고 25분)</span>

오늘 경험한 사이클을 팀과 조직으로 확장하는 그림을 그립니다.
우아한형제들 기술 블로그 사례 3편과 연결하고, 액션플랜을 작성합니다.

[반복에서 축적으로 →](/workshop/advanced/accumulation)

</Steps>
