# autoresearch 루프

## Karpathy의 autoresearch

2026년 3월, Andrej Karpathy가 [autoresearch](https://github.com/karpathy/autoresearch)를 공개합니다.

<Callout type="default">
**autoresearch의 핵심 아이디어**

AI 에이전트에게 작은 LLM 학습 환경을 주고, 밤새 자율적으로 실험하게 한다.
코드를 수정하고, 5분간 학습하고, 결과가 나아졌으면 유지하고, 아니면 버리고, 반복한다.
아침에 일어나면 실험 로그와 (운이 좋으면) 더 나은 모델이 있다.
</Callout>

파일이 세 개뿐입니다:

| 파일 | 역할 | 누가 건드리나 |
|------|------|-------------|
| `prepare.py` | 데이터 준비, 평가 함수 | 아무도 안 건드림 (고정) |
| `train.py` | 모델, 옵티마이저, 학습 루프 | **AI가 수정** |
| `program.md` | AI에게 주는 지시서 | **사람이 다듬음** |

사람은 `train.py`를 직접 건드리지 않습니다.
오직 `program.md`만 다듬습니다.

Karpathy는 이 `.md` 파일을 이렇게 표현합니다:

> *"The `program.md` file is essentially a super lightweight skill."*

## 루프의 구조

```
program.md (사람이 작성)
    ↓
AI가 train.py 수정
    ↓
5분간 학습
    ↓
val_bpb 측정 (낮을수록 좋음)
    ↓
결과 보고 → program.md 개선
    ↓
(반복)
```

밤새 돌리면 약 100번의 실험이 진행됩니다.
매 실험마다 `val_bpb`가 측정되고, 좋은 결과는 유지됩니다.

핵심은 세 가지입니다:

1. **하나의 파일**만 AI가 수정한다 (`train.py`)
2. **하나의 지표**로 평가한다 (`val_bpb`)
3. **고정된 시간 예산** 안에서 반복한다 (5분)

## 업무에 적용하면?

같은 구조를 우리 업무에 대입합니다.

| autoresearch | 오늘 워크숍 |
|---|---|
| `train.py` (AI가 수정하는 코드) | 업무 결과물 (AI가 생성하는 코드리뷰, 문서, TC) |
| `program.md` (사람이 다듬는 지시서) | <strong>내 업무 지침서</strong> (ChatGPT 커스텀 지침 / Gemini Gems) |
| `val_bpb` (단일 평가 지표) | <strong>체크리스트 점수</strong> (팀 컨벤션 부합도, 누락 항목 수 등) |
| 5분 고정 시간 예산 | <strong>~15분 1사이클</strong> |
| 밤새 100번 자동 반복 | <strong>3사이클 직접 반복</strong> |

```
내 지침서 (사람이 작성)
    ↓
AI가 업무 결과물 생성
    ↓
평가 기준으로 채점
    ↓
개선 포인트 메모 → 지침서 수정
    ↓
(반복)
```

<Callout type="info">
**핵심 메시지**

autoresearch에서 사람이 프로그래밍하는 대상은 Python이 아니라 `program.md`입니다.
오늘 워크숍에서 여러분이 프로그래밍하는 대상은 코드가 아니라 <strong>업무 지침서</strong>입니다.
</Callout>

## 우아한형제들 사례: 왜 구조를 줘야 하는가

이 루프가 왜 작동하는지, 우아한형제들의 실제 사례가 보여줍니다.

### AI와 함께하는 테스트 자동화: 플러그인 개발기

> 출처: [우아한형제들 기술블로그](https://techblog.woowahan.com/24568/) (강규한, 2025.12)

커머스 백엔드 개발자가 AI로 테스트 코드를 만들려 했습니다.

**1차 시도: AI에게 "테스트 코드 작성해줘"**
- 10개 클래스 테스트 → 컴파일 성공률 <strong>10%</strong> (9/10 실패)
- import 오타, 존재하지 않는 클래스 참조, 타입 불일치

**2차 시도: 구조를 먼저 제공 + AI는 구현만**
- IntelliJ 플러그인이 코드를 분석해서 TODO 템플릿을 만들고
- AI는 그 템플릿 안의 TODO만 구현
- 컴파일 성공률 <strong>95%</strong>, 작업 시간 10분 → 3분

| 지표 | 이전 | 이후 |
|------|------|------|
| 클래스당 작업 시간 | 10분 | 3분 (70% 단축) |
| 컴파일 성공률 | 85% | 95% |
| 메서드 커버리지 | 67% | 95% |

<Callout type="warning">
**이것이 오늘 워크숍에서 지침서를 만드는 이유입니다.**

"코드 리뷰해줘"라고만 하면 (= 빈 지침서) → 일반적이고 뻔한 결과
"이 구조와 기준으로 리뷰해줘"라고 하면 (= 잘 쓴 지침서) → 전문가 수준의 결과

테스트 자동화 플러그인이 성공한 것과 같은 원리입니다.
<strong>AI의 한계를 인정하고, 구조를 줘야 합니다.</strong>
</Callout>
