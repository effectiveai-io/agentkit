# autoresearch란

## 배경: Andrej Karpathy와 에이전틱 엔지니어링

[Andrej Karpathy](https://karpathy.ai/)는 OpenAI 공동창업자이자 Tesla AI 디렉터 출신으로, AI 연구와 교육 분야에서 가장 영향력 있는 인물 중 한 명입니다.

2025년 2월, Karpathy는 <strong>"바이브 코딩"(Vibe Coding)</strong>이라는 용어를 만들었습니다 — "분위기에 맡겨서 AI에게 코딩을 시키는" 접근이었습니다.

1년 뒤인 2026년 2월, Karpathy는 바이브 코딩 1주년을 회고하며 <strong>"에이전틱 엔지니어링"(Agentic Engineering)</strong>이라는 새로운 개념을 제안합니다:

> *"'Agentic' — 99%의 시간 동안 코드를 직접 쓰지 않고, 에이전트를 오케스트레이션하며 감독하는 것이다.
> 'Engineering' — 거기에는 기술과 과학, 전문성이 있다. 배울 수 있고, 더 잘할 수 있는 것이다."*
>
> — [Andrej Karpathy, 바이브 코딩 1주년 회고 (2026.02)](https://x.com/karpathy/status/2019137879310836075)

그리고 2026년 3월, 이 철학을 **행동으로 보여주는** 프로젝트를 공개합니다.

## autoresearch

[autoresearch](https://github.com/karpathy/autoresearch)는 Karpathy가 2026년 3월에 공개한 프로젝트입니다.
공개 3주 만에 GitHub 스타 50,000개를 넘었고, 다양한 플랫폼(macOS, Windows, AMD)으로 포크되며 커뮤니티의 큰 반향을 일으켰습니다.

<Callout type="default">
**Karpathy의 설명**

*"AI 에이전트에게 작은 LLM 학습 환경을 주고 밤새 자율적으로 실험하게 한다.
사람은 코드를 건드리지 않는다. 대신 `program.md` 마크다운 파일을 프로그래밍한다."*

*"The core idea is that you're not touching any of the Python files like you normally would as a researcher. Instead, you are programming the `program.md` Markdown files."*

— [autoresearch README](https://github.com/karpathy/autoresearch)
</Callout>

### 왜 이게 중요한가

autoresearch는 단순한 실험 도구가 아니라, **"사람이 일하는 방식의 변화"**를 보여줍니다:

- <strong>이전</strong>: 사람이 코드를 직접 작성하고 실험하고 측정
- <strong>이후</strong>: 사람은 지시서(`program.md`)만 다듬고, AI가 코드 수정 → 실험 → 측정 → 판단을 자율적으로 반복

이 워크숍에서는 같은 원리를 **여러분의 업무**에 적용합니다.

---

## 핵심 구조

파일이 세 개뿐입니다:

| 파일 | 역할 | 누가 건드리나 |
|------|------|-------------|
| `prepare.py` | 데이터 준비, 평가 함수 | 아무도 안 건드림 (고정) |
| `train.py` | 모델, 옵티마이저, 학습 루프 | **AI가 수정** |
| `program.md` | AI에게 주는 지시서 | **사람이 다듬음** |

사람은 `train.py`를 직접 건드리지 않습니다.
오직 `program.md`만 다듬습니다.

Karpathy는 이 `.md` 파일을 이렇게 표현합니다:

> *"The `program.md` file is essentially a super lightweight skill."*

## 루프의 구조

```
program.md (사람이 작성)
    ↓
AI가 train.py 수정
    ↓
5분간 학습
    ↓
val_bpb 측정 (낮을수록 좋음)
    ↓
결과 보고 → program.md 개선
    ↓
(반복)
```

밤새 돌리면 약 100번의 실험이 진행됩니다.
매 실험마다 `val_bpb`가 측정되고, 좋은 결과는 유지됩니다.

핵심은 세 가지입니다:

1. **하나의 파일**만 AI가 수정한다 (`train.py`)
2. **하나의 지표**로 평가한다 (`val_bpb`)
3. **고정된 시간 예산** 안에서 반복한다 (5분)

## 업무에 적용하면?

같은 구조를 우리 업무에 대입합니다.

| autoresearch | 오늘 워크숍 |
|---|---|
| `train.py` (AI가 수정하는 코드) | 업무 결과물 (AI가 생성하는 코드리뷰, 문서, TC) |
| `program.md` (사람이 다듬는 지시서) | <strong>내 업무 프롬프트</strong> (GitHub 저장소에 관리) |
| `val_bpb` (단일 평가 지표) | <strong>체크리스트 점수</strong> (0~10점) |
| 5분 고정 시간 예산 | AI가 자동으로 3사이클 |
| 밤새 100번 자동 반복 | Gemini가 자동 개선 루프 실행 |

```
내 프롬프트 (사람이 작성)
    ↓
AI가 업무 결과물 생성
    ↓
평가 기준으로 자체 채점
    ↓
AI가 프롬프트 개선안 도출
    ↓
(자동 반복)
```

<Callout type="info">
**핵심 메시지**

autoresearch에서 사람이 프로그래밍하는 대상은 Python이 아니라 `program.md`입니다.
오늘 워크숍에서 여러분이 프로그래밍하는 대상은 코드가 아니라 <strong>업무 프롬프트</strong>입니다.
</Callout>

## 우아한형제들 사례: 왜 구조를 줘야 하는가

이 루프가 왜 작동하는지, 우아한형제들의 실제 사례가 보여줍니다.

### AI와 함께하는 테스트 자동화: 플러그인 개발기

> 출처: [우아한형제들 기술블로그](https://techblog.woowahan.com/24568/) (강규한, 2025.12)

커머스 백엔드 개발자가 AI로 테스트 코드를 만들려 했습니다.

**1차 시도: AI에게 "테스트 코드 작성해줘"**
- 10개 클래스 테스트 → 컴파일 성공률 <strong>10%</strong> (9/10 실패)
- import 오타, 존재하지 않는 클래스 참조, 타입 불일치

**2차 시도: 구조를 먼저 제공 + AI는 구현만**
- IntelliJ 플러그인이 코드를 분석해서 TODO 템플릿을 만들고
- AI는 그 템플릿 안의 TODO만 구현
- 컴파일 성공률 <strong>95%</strong>, 작업 시간 10분 → 3분

| 지표 | 이전 | 이후 |
|------|------|------|
| 클래스당 작업 시간 | 10분 | 3분 (70% 단축) |
| 컴파일 성공률 | 85% | 95% |
| 메서드 커버리지 | 67% | 95% |

<Callout type="warning">
**이것이 오늘 워크숍에서 프롬프트를 다듬는 이유입니다.**

"코드 리뷰해줘"라고만 하면 (= 빈 프롬프트) → 일반적이고 뻔한 결과.
"이 구조와 기준으로 리뷰해줘"라고 하면 (= 잘 쓴 프롬프트) → 전문가 수준의 결과.

테스트 자동화 플러그인이 성공한 것과 같은 원리입니다.
<strong>AI의 한계를 인정하고, 구조를 줘야 합니다.</strong>
</Callout>

## program.md 전문

autoresearch의 핵심인 `program.md` 전문입니다.
사람이 AI에게 주는 유일한 지시서 — 이것이 autoresearch의 전부입니다.

<Tabs items={['한국어 번역', '영어 원문']}>
<Tabs.Tab>

```markdown
# autoresearch

LLM이 스스로 연구를 수행하는 실험입니다.

## 셋업

새 실험을 시작할 때, 사용자와 함께 다음을 진행합니다:

1. **실행 태그 합의**: 오늘 날짜 기반 태그를 제안합니다 (예: `mar5`).
   `autoresearch/<태그>` 브랜치가 이미 존재하면 안 됩니다 — 새로운 실행입니다.
2. **브랜치 생성**: 현재 master에서 `git checkout -b autoresearch/<태그>`.
3. **범위 내 파일 읽기**: 레포가 작습니다. 전체 맥락을 위해 다음 파일을 읽으세요:
   - `README.md` — 저장소 맥락.
   - `prepare.py` — 고정 상수, 데이터 준비, 토크나이저, 데이터로더, 평가. 수정 금지.
   - `train.py` — 당신이 수정하는 파일. 모델 아키텍처, 옵티마이저, 학습 루프.
4. **데이터 존재 확인**: `~/.cache/autoresearch/`에 데이터 샤드와 토크나이저가 있는지 확인.
   없으면 사람에게 `uv run prepare.py` 실행을 요청합니다.
5. **results.tsv 초기화**: 헤더 행만 있는 `results.tsv`를 생성합니다.
   베이스라인은 첫 번째 실행 후 기록됩니다.
6. **확인 후 시작**: 셋업이 정상인지 확인합니다.

확인을 받으면 실험을 시작합니다.

## 실험

각 실험은 단일 GPU에서 실행됩니다. 학습 스크립트는 **고정 시간 예산 5분**(벽시계 학습
시간, 시작/컴파일 제외)으로 실행됩니다. 실행 방법: `uv run train.py`.

**할 수 있는 것:**
- `train.py` 수정 — 이것만 편집합니다. 모든 것이 대상입니다: 모델 아키텍처, 옵티마이저,
  하이퍼파라미터, 학습 루프, 배치 크기, 모델 크기 등.

**할 수 없는 것:**
- `prepare.py` 수정. 읽기 전용입니다.
- 새 패키지 설치나 의존성 추가. `pyproject.toml`에 있는 것만 사용 가능합니다.
- 평가 코드 수정. `prepare.py`의 `evaluate_bpb` 함수가 ground truth 지표입니다.

**목표는 단순합니다: 가장 낮은 val_bpb를 달성하는 것.**

**단순성 기준**: 다른 조건이 같다면 단순한 것이 낫습니다.
복잡성을 추가하는 작은 개선은 가치가 없습니다.
코드를 삭제해서 같거나 더 나은 결과를 얻으면 — 반드시 유지.

## 실험 루프

영원히 반복:

1. git 상태 확인: 현재 브랜치/커밋
2. 실험 아이디어로 `train.py`를 직접 수정
3. git 커밋
4. 실험 실행: `uv run train.py > run.log 2>&1`
5. 결과 읽기: `grep "^val_bpb:" run.log`
6. 크래시 시 스택 트레이스 읽고 수정 시도
7. TSV에 결과 기록
8. val_bpb가 개선되면 브랜치 전진, 아니면 git reset

**절대 멈추지 마세요**: 실험 루프가 시작되면 사람에게 계속할지 묻지 마세요.
사람은 자고 있을 수 있습니다. 당신은 자율적입니다.
루프는 사람이 중단할 때까지 실행됩니다.

사용 예시: 사용자가 잠자는 동안 시간당 약 12회, 수면 중 약 100회 실험.
아침에 일어나면 결과가 모두 완료되어 있습니다!
```

```markdown
# autoresearch

This is an experiment to have the LLM do its own research.

## Setup

To set up a new experiment, work with the user to:

1. **Agree on a run tag**: propose a tag based on today's date (e.g. `mar5`).
   The branch `autoresearch/<tag>` must not already exist — this is a fresh run.
2. **Create the branch**: `git checkout -b autoresearch/<tag>` from current master.
3. **Read the in-scope files**: The repo is small. Read these files for full context:
   - `README.md` — repository context.
   - `prepare.py` — fixed constants, data prep, tokenizer, dataloader, evaluation.
     Do not modify.
   - `train.py` — the file you modify. Model architecture, optimizer, training loop.
4. **Verify data exists**: Check that `~/.cache/autoresearch/` contains data shards
   and a tokenizer. If not, tell the human to run `uv run prepare.py`.
5. **Initialize results.tsv**: Create `results.tsv` with just the header row.
6. **Confirm and go**: Confirm setup looks good.

## Experimentation

Each experiment runs on a single GPU. The training script runs for a **fixed time
budget of 5 minutes** (wall clock training time, excluding startup/compilation).

**What you CAN do:**
- Modify `train.py` — this is the only file you edit.

**What you CANNOT do:**
- Modify `prepare.py`. It is read-only.
- Install new packages or add dependencies.
- Modify the evaluation harness.

**The goal is simple: get the lowest val_bpb.**

**Simplicity criterion**: All else being equal, simpler is better.
A 0.001 improvement that adds 20 lines of hacky code? Probably not worth it.
A 0.001 improvement from deleting code? Definitely keep.

## The experiment loop

LOOP FOREVER:

1. Look at the git state
2. Tune `train.py` with an experimental idea
3. git commit
4. Run the experiment: `uv run train.py > run.log 2>&1`
5. Read out the results: `grep "^val_bpb:" run.log`
6. If crash, read stack trace and attempt fix
7. Record results in tsv
8. If improved, advance branch. If not, git reset.

**NEVER STOP**: Once the experiment loop has begun, do NOT pause to ask the
human if you should continue. The human might be asleep. You are autonomous.

As an example, a user might leave you running while they sleep.
~12 experiments/hour, ~100 over a night's sleep.
```
